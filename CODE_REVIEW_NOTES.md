# Code Review Notes — BookRafina

Ημερομηνία review: 2026-03-16

## Γενική εικόνα
Το πρόγραμμα είναι καλοστημένο για single-file offline εφαρμογή. Δεν εντοπίστηκαν syntax errors στο `syllógos.html`. Η οργάνωση σε `FileStorage`, `Store`, `Utils`, `Views` είναι σωστή και το app είναι ήδη λειτουργικό.

---

## Κύριες παρατηρήσεις

### 1. Δεν είναι 100% offline ακόμα
Το app φορτώνει εξωτερικά assets:
- Google Fonts
- SheetJS από CDN
- Google Fonts ξανά στο printable receipt

**Κίνδυνος:** χωρίς internet μπορεί να μην εμφανίζονται σωστά γραμματοσειρές και να μη δουλεύει το Excel export.

**Πρόταση:**
- αποθήκευση του `xlsx.full.min.js` τοπικά στο ίδιο folder ή σε υποφάκελο
- αφαίρεση των external fonts ή χρήση system fonts / local fonts

---

### 2. Το σύστημα αποδείξεων μπορεί να ξαναδώσει αριθμό
Ο επόμενος αριθμός απόδειξης υπολογίζεται από τα υπάρχοντα payments. Αν διαγραφεί πληρωμή, ειδικά η τελευταία, υπάρχει κίνδυνος να ξαναχρησιμοποιηθεί αριθμός.

**Κίνδυνος:** λογιστική ασυνέπεια σε σχέση με φυσικό μπλοκάκι αποδείξεων.

**Πρόταση:**
- αποθήκευση ξεχωριστού μετρητή ανά έτος, π.χ. `lastReceiptNumberByYear`
- ο μετρητής να αυξάνει μόνο προς τα πάνω, ποτέ να μην υπολογίζεται ξανά από τα υπάρχοντα δεδομένα

---

### 3. Τα υπάρχοντα payments στο JSON δεν έχουν ακόμα `receiptNumber`
Στο τρέχον `syllogos_data.json` τα παλιά payments δεν έχουν αποθηκευμένο αριθμό απόδειξης.

**Κίνδυνος:** οι αριθμοί εμφανίζονται δυναμικά αλλά δεν έχουν γίνει πραγματικό migration στα δεδομένα.

**Πρόταση:**
- κατά την πρώτη φόρτωση να γίνεται migration
- να συμπληρώνονται `receiptNumber` στα παλιά payments
- να γίνεται άμεσο save του ενημερωμένου αρχείου

---

### 4. Ασυνέπεια ανάμεσα σε payment και receipt
Η αποθήκευση γίνεται με μία εγγραφή ανά μήνα, αλλά η εκτύπωση receipt μπορεί να δείχνει πολλά payments μαζί σαν ένα αποδεικτικό.

**Κίνδυνος:** μπερδεμένο μοντέλο για χρήστη/λογιστή.

**Πρόταση:**
Να επιλεγεί ένα σαφές μοντέλο:
- είτε 1 payment = 1 receipt
- είτε 1 payment = πολλοί μήνες σε μία εγγραφή

---

### 5. Η εισαγωγή backup δεν είναι ατομική
Κατά το import γίνονται διαδοχικά saves σε members, payments, config.

**Κίνδυνος:** αν αποτύχει κάτι στη μέση, το σύστημα μπορεί να μείνει σε μερικώς ενημερωμένη κατάσταση.

**Πρόταση:**
- πρώτα πλήρες validation όλου του αρχείου
- μετά ενιαία εφαρμογή των δεδομένων
- ιδανικά με temporary object και μόνο ένα τελικό commit

---

### 6. Το validation του JSON αρχείου είναι περιορισμένο
Γίνεται βασικός έλεγχος ότι `members` και `payments` είναι arrays, αλλά όχι ουσιαστικός έλεγχος schema.

**Κίνδυνος:** ένα λάθος ή πειραγμένο JSON μπορεί να φορτωθεί και να δημιουργήσει αστάθεια.

**Πρόταση:**
Έλεγχος για:
- `version`
- required keys
- valid `year`, `month`, `amount`
- έγκυρο `memberId`
- σωστές δομές για `config` και `categories`

---

## Μεσαίας προτεραιότητας παρατηρήσεις

### 7. Πιθανό θέμα ημερομηνίας λόγω UTC
Η παραγωγή ISO date με `toISOString()` μπορεί κοντά στα μεσάνυχτα να δώσει λάθος μέρα.

**Πρόταση:**
- χρήση local formatter με `getFullYear()`, `getMonth() + 1`, `getDate()`

---

### 8. Plain JSON με προσωπικά δεδομένα
Το αρχείο περιέχει τηλέφωνα, email, αριθμούς ταυτότητας και λοιπά προσωπικά δεδομένα χωρίς κρυπτογράφηση.

**Κίνδυνος:** αν αντιγραφεί το αρχείο, τα δεδομένα διαβάζονται άμεσα.

**Πρόταση:**
- σαφής προειδοποίηση στον χρήστη
- ασφαλής αποθήκευση / backup
- μελλοντικά προαιρετικό κρυπτογραφημένο backup

---

### 9. Τα filenames για export θέλουν sanitization
Το όνομα του συλλόγου χρησιμοποιείται σε filenames export.

**Κίνδυνος:** ειδικοί χαρακτήρες όπως `/`, `:`, `\\` μπορεί να δημιουργήσουν πρόβλημα σε save/export.

**Πρόταση:**
- sanitize του filename πριν το export

---

### 10. Το file workflow εξαρτάται από Chrome/Edge
Το File System Access API δεν υποστηρίζεται παντού.

**Πρόταση:**
- πιο σαφές μήνυμα συμβατότητας στην αρχή
- fallback με manual backup/export όπου χρειάζεται

---

### 11. Χρέωση από ημερομηνία εγγραφής
Η λογική χρέωσης φαίνεται να αγνοεί μήνα αν η εγγραφή έγινε μετά την 1η του μήνα.

**Ερώτημα επιχειρησιακής λογικής:**
Αν ένα μέλος γράφτηκε 15/01, πρέπει να χρεώνεται για Ιανουάριο ή όχι;

**Πρόταση:**
- να επιβεβαιωθεί ο κανόνας
- ίσως να προστεθεί ρύθμιση για το πότε ξεκινά η χρέωση

---

## Προτεινόμενη σειρά υλοποίησης αργότερα
1. Πλήρως local assets για πραγματικό offline mode
2. Μόνιμος yearly counter για αποδείξεις
3. Migration παλιών payments με `receiptNumber`
4. Πιο αυστηρό validation/import schema
5. Local-safe formatting ημερομηνιών
6. Sanitize στα export filenames
7. Βελτίωση πολιτικής backup / ιδιωτικότητας

---

## Συμπέρασμα
Η βάση του έργου είναι καλή και ήδη χρήσιμη για καθημερινή χρήση. Οι βασικές βελτιώσεις αφορούν κυρίως:
- αξιοπιστία offline λειτουργίας
- λογιστική συνέπεια στις αποδείξεις
- ανθεκτικότητα του import/export
- καλύτερη διαχείριση προσωπικών δεδομένων
